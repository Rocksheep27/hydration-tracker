# HydrationTracker Roadmap

## 总体方向

HydrationTracker 最终希望成为一个可以在 iPhone 上独立使用的个人水分摄入管理工具。

项目已经通过 V1 验证记录和计算逻辑，并在 V2 完成 Mac 本地 Web App。V3 将建立不依赖 Mac 后端的纯前端 PWA，让 iPhone 可以通过 Safari 独立使用并优先把数据保存在手机本地。

暂时不做：
- iOS 原生 App
- Swift 开发
- App Store 配置

## V1：核心计算逻辑原型（已完成）

目标：验证记录、估算、汇总和本地保存逻辑是否清楚、可用。

已实现：
- 可以在 Mac 终端运行。
- 记录白水、饮品和食物。
- 提供常用食物和饮品库。
- 为常用项目提供默认单位及估算重量或容量。
- 根据含水率估算水分摄入。
- 查看今日进度、目标完成率和剩余量。
- 查看今日记录明细。
- 删除今日单条记录。
- 清空今日记录，同时保留其他日期数据。
- 使用本地 JSON 文件保存数据。

当前限制：
- 仍是 Python 命令行原型，不是最终 App。
- 每日目标暂时使用程序内置的默认值 `2000 ml`。
- 自定义每日目标留到后续增强阶段再考虑。
- 只使用 Python 标准库，不需要安装额外依赖。

## V2：本地 Web App（已完成）

目标：把核心逻辑做成浏览器界面，方便日常输入。

完成阶段：

- V2.1：只读今日概览，显示总量、完成率、剩余量和记录数量。
- V2.2：网页添加白水、饮品和食物记录，保存后自动刷新概览。
- V2.3：网页查看今日明细、删除单条记录和清空今日记录。

当前成果：

- 使用 Python 标准库提供 Mac 本地 Web App。
- 复用 V1 的常用项目库、水分计算和 JSON 读写逻辑。
- 支持网页新增、查看和整理今日记录。
- 单条删除和清空今日都要求二次确认。
- 删除和清空只作用于今天，不影响其他日期记录。
- 数据只保存在本地 JSON，不联网、不上传。
- V1 命令行版本保持可用。

实现方式：
- 使用标准库 `http.server` 和原生 HTML/CSS/JavaScript。
- 保持计算逻辑和界面展示逻辑分开。

## V3：iPhone 独立 PWA 原型

目标：让 iPhone 可以独立完成日常记录，不依赖 Mac 开机、同一局域网、同一 Apple ID、iCloud、账号或云同步。Mac 继续负责开发、测试和发布更新。

核心路线：

- 使用纯前端 HTML、CSS 和 JavaScript。
- 日常运行不依赖 Python 后端。
- 把常用项目库和水分计算逻辑迁移到浏览器端。
- 使用浏览器本地存储保存 iPhone 记录。
- 支持 Safari、主屏幕安装和基础离线使用。
- 使用固定 HTTPS 网址发布静态应用，程序更新只更新网页代码。
- 使用稳定存储 key、`schema_version` 和迁移逻辑保留升级前数据。
- 提供 JSON 导入导出，用于备份、换手机和从 Mac 迁移数据。

阶段计划：

- V3.1：PWA 方案设计与路线图调整。
- V3.2：创建 `pwa/` 纯前端最小原型（已完成）。
- V3.3：迁移常用项目库和水分计算逻辑到浏览器端（已完成）。
- V3.4：使用 localStorage 保存、读取、删除和清空记录（已完成）。
- V3.5：适配 iPhone Safari 布局和触控体验（已完成）。
- V3.6：添加 manifest、图标和基础离线缓存能力（已完成）。
- V3.7：实现 JSON 导入导出，用于备份和迁移（已完成）。
- V3.8：历史记录查看与历史数据管理（已完成）。
- V3.9：固定 HTTPS 发布方式研究（已开始，尚未正式发布）。

实现方式：
- V3 原型优先使用 localStorage；数据量和查询复杂度增加后再评估 IndexedDB。
- Mac 本地预览只用于开发测试，不作为 iPhone 日常运行方式。
- 日常使用推荐通过固定静态托管网址访问，用户数据仍保存在 iPhone 本地。
- 暂时不做 iOS 原生 App。
- 暂时不使用 Swift。
- 暂时不做 App Store 配置。

## V4：增强功能

目标：增加长期使用所需的辅助功能。

计划内容：
- 喝水提醒通知。
- 在 V3.8 基础上扩展更复杂的历史统计和趋势分析。
- 可管理的常用饮料模板。
- 可管理的常用食物模板。
- 每日目标设置。
- 可选旧数据清理：
  - 当前按日期查看、补录、单条删除和清空指定日期已经满足使用需求。
  - 暂不实现批量清理；未来如确有需要，再考虑删除 30 天前、90 天前或指定日期以前的记录。
  - 所有批量删除操作必须明确显示范围并要求二次确认。
- 扩展常用项目库：
  - 用户想到新的常吃食物或饮品时，可以继续逐步补充。
  - 每个新增项目包含 `id`、`name`、`category`、`default_unit`、`default_amount`、`base_type`、`water_ratio` 和 `emoji`。
  - 新增项目不得破坏旧记录；旧记录继续根据 `item_id` 或 `name` 显示。
  - 这类补充可以作为后续 V3 或 V4 的小版本更新。
- 自定义食物和饮品记录：
  - 用户可以输入不在常用项目库中的名称、单位、数量、估算水分和备注。
  - 方式一是直接输入估算水分，例如“火锅 1 份，估算 500 ml 水分”。
  - 方式二是输入重量或容量及含水率，由程序估算水分。
  - 自定义记录不改变常用项目库；“保存为常用项目”需要单独设计。
  - 建议放在 V4，或至少等 V3.7 JSON 导入导出完成后再实现，以便备份和迁移自定义数据。
- 按时间进度判断补水建议：
  - 根据当前时间、每日目标和饮水计划时间段，估算当前理论应摄入水分。
  - 对比实际摄入，判断当前进度是否落后。
  - 如果落后，显示建议补水量。
  - 默认饮水时间段可先设为 8:00–22:00。
  - 未来支持自定义饮水时间段。
  - 该功能不在 V3 实现，留到 V4 或与提醒模块一起开发。
- 更清晰的趋势展示。

## 当前阶段

V1 和 V2 均已完成并正式收尾。V3.1 至 V3.8 已完成。V3.8 已通过真实 iPhone Safari 测试，PWA 现分为“今日 / 历史 / 备份”三个视图；历史页支持月历、月份切换、指定日期明细、近 7 日趋势图、指定日期补录、历史单条删除和清空指定日期。

当前历史记录继续保存在 `hydration_tracker_v3_records` 的 `records` 数组中。旧数据清理暂不实现，已移到未来可选增强，不作为 V3.8 必做项。V3.9 已开始固定 HTTPS 发布方式研究，本阶段只比较方案、确定数据迁移边界和整理验证清单，尚未实际发布。V4 保留提醒、复杂历史统计、每日目标设置、按时间进度补水建议、自定义记录及其他增强功能。

V3.9 开始前需要注意：

1. 比较 GitHub Pages、Netlify、Vercel 等静态托管方式，优先选择稳定、简单的固定 HTTPS 来源。
2. 保持网址来源稳定；协议、域名或端口变化会影响浏览器读取原有 localStorage。
3. 发布前先导出 JSON 备份，并验证导入恢复流程。
4. 固定 HTTPS 发布后，完整验证 Service Worker 更新、离线启动和主屏幕 Web App。
5. 继续保持 `hydration_tracker_v3_records`、`schema_version: 1`、JSON 导入导出格式和 Service Worker 用户数据边界不变。
6. 明确每台设备的数据保存在各自浏览器本地，不提供账号、云同步或自动共享。

### V3.8：历史记录查看与历史数据管理（已完成）

- 页面结构重组已完成：PWA 分为“今日 / 历史 / 备份”三个视图。
- 历史月历、月份切换、指定日期明细和近 7 日 SVG 趋势图已完成。
- 页面结构已通过真实 iPhone Safari 手动测试。
- 历史单条删除已完成：按日期和记录 `id` 精确匹配，删除前二次确认。
- 指定日期补录已完成并通过真实 iPhone Safari 手动测试：写入月历选中日期和用户填写时间。
- 清空指定日期已完成：按选中日期精确清空，确认文案显示日期和记录数量。
- 旧数据清理暂不实现，已移到未来可选增强。
- 所有历史管理继续使用 `hydration_tracker_v3_records`，不改变现有记录来源。
- 默认保持 `schema_version: 1`，不调用 `localStorage.clear()`。
- 不改变 JSON 导入导出格式，也不让 Service Worker 读取、缓存或修改用户记录。

### V3.9：GitHub Pages 发布准备（已开始）

- 方案研究已完成，当前进入 GitHub Pages 发布配置阶段。
- 已比较 GitHub Pages、Netlify、Vercel 和 Mac 局域网临时服务器。
- 推荐优先使用 GitHub Pages，并通过 GitHub Actions 只发布 `pwa/` 静态文件，避免 Pages 产物包含用户 JSON 和其他开发文件。
- GitHub Pages 默认项目网址可作为固定 HTTPS 来源；如以后更换域名或平台，需要通过 JSON 导出和导入迁移数据。
- 发布前必须从当前实际使用环境导出 JSON 备份，并确认 `hydration_tracker_v3_records`、`schema_version: 1` 和备份格式保持不变。
- 发布后需要验证主屏幕安装、同一来源数据保留、Service Worker 更新、离线启动、JSON 恢复及家人设备独立使用。
- Service Worker 继续只缓存静态资源，用户记录仍只保存在各设备 localStorage。
- 本地 Git 初始化、`main` 分支、`origin` 设置、首次本地提交和首次推送已完成。
- 发布目标已确定为 GitHub 用户 `Rocksheep27` 的 `hydration-tracker` 仓库，预计网址为 `https://rocksheep27.github.io/hydration-tracker/`。
- 上传前必须通过 `.gitignore` 排除 `data/hydration_log.json`、JSON 备份、环境文件和本地缓存。
- PWA 的相对资源、manifest、Service Worker 注册和缓存路径适合 `/hydration-tracker/` 子路径。
- GitHub Actions Pages workflow 已进入配置阶段，将只发布 `pwa/`，避免 Pages 产物包含 V1/V2 文件或本地数据。
- 下一步是把 `.github/workflows/deploy-pages.yml` 推送到 GitHub，然后在 GitHub 网页端把 Pages Source 设置为 GitHub Actions。
