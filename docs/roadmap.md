# HydrationTracker Roadmap

## 总体方向

HydrationTracker 最终希望成为一个可以在 iPhone 上独立使用的个人水分摄入管理工具。

项目已经通过 V1 验证记录和计算逻辑，并在 V2 完成 Mac 本地 Web App。V3 已完成固定 HTTPS 发布的纯前端 PWA，让 iPhone 可以通过 Safari 独立使用并优先把数据保存在手机本地。

暂时不做：
- iOS 原生 App
- Swift 开发
- App Store 配置

## V1：核心计算逻辑原型（已完成）

目标：验证记录、估算、汇总和本地保存逻辑是否清楚、可用。

已实现：
- 可以在 Mac 终端运行。
- 记录白水、饮品和食物。
- 提供常用食物和饮品库。
- 为常用项目提供默认单位及估算重量或容量。
- 根据含水率估算水分摄入。
- 查看今日进度、目标完成率和剩余量。
- 查看今日记录明细。
- 删除今日单条记录。
- 清空今日记录，同时保留其他日期数据。
- 使用本地 JSON 文件保存数据。

当前限制：
- 仍是 Python 命令行原型，不是最终 App。
- 每日目标暂时使用程序内置的默认值 `2000 ml`。
- 自定义每日目标留到后续增强阶段再考虑。
- 只使用 Python 标准库，不需要安装额外依赖。

## V2：本地 Web App（已完成）

目标：把核心逻辑做成浏览器界面，方便日常输入。

完成阶段：
- V2.1：只读今日概览，显示总量、完成率、剩余量和记录数量。
- V2.2：网页添加白水、饮品和食物记录，保存后自动刷新概览。
- V2.3：网页查看今日明细、删除单条记录和清空今日记录。

当前成果：
- 使用 Python 标准库提供 Mac 本地 Web App。
- 复用 V1 的常用项目库、水分计算和 JSON 读写逻辑。
- 支持网页新增、查看和整理今日记录。
- 单条删除和清空今日都要求二次确认。
- 删除和清空只作用于今天，不影响其他日期记录。
- 数据只保存在本地 JSON，不联网、不上传。
- V1 命令行版本保持可用。

实现方式：
- 使用标准库 `http.server` 和原生 HTML/CSS/JavaScript。
- 保持计算逻辑和界面展示逻辑分开。

## V3：iPhone 独立 PWA 原型（已完成）

目标：让 iPhone 可以独立完成日常记录，不依赖 Mac 开机、同一局域网、同一 Apple ID、iCloud、账号或云同步。Mac 继续负责开发、测试和发布更新。

核心路线：
- 使用纯前端 HTML、CSS 和 JavaScript。
- 日常运行不依赖 Python 后端。
- 把常用项目库和水分计算逻辑迁移到浏览器端。
- 使用浏览器本地存储保存 iPhone 记录。
- 支持 Safari、主屏幕安装和基础离线使用。
- 使用固定 HTTPS 网址发布静态应用，程序更新只更新网页代码。
- 使用稳定存储 key、`schema_version` 和迁移逻辑保留升级前数据。
- 提供 JSON 导入导出，用于备份、换手机和从 Mac 迁移数据。

阶段计划：
- V3.1：PWA 方案设计与路线图调整（已完成）。
- V3.2：创建 `pwa/` 纯前端最小原型（已完成）。
- V3.3：迁移常用项目库和水分计算逻辑到浏览器端（已完成）。
- V3.4：使用 localStorage 保存、读取、删除和清空记录（已完成）。
- V3.5：适配 iPhone Safari 布局和触控体验（已完成）。
- V3.6：添加 manifest、图标和基础离线缓存能力（已完成）。
- V3.7：实现 JSON 导入导出，用于备份和迁移（已完成）。
- V3.8：历史记录查看与历史数据管理（已完成）。
- V3.9：固定 HTTPS 发布方式研究与 GitHub Pages 发布（已完成）。

当前成果：
- 已通过 GitHub Actions 将 PWA 发布到 `https://rocksheep27.github.io/hydration-tracker/`。
- PWA 现分为“今日 / 历史 / 备份”三个视图。
- 今日页支持添加记录、查看明细、删除单条和清空今日。
- 历史页支持月历、月份切换、指定日期明细、近 7 日趋势、指定日期补录、历史单条删除和清空指定日期。
- 备份页支持 JSON 导出和导入。
- 用户记录继续保存在各自设备本地 localStorage 中，不需要账号，也不自动同步。
- Service Worker 继续只缓存静态资源，不操作用户记录数据。
- V3 至此已达到“可长期使用的个人 PWA 原型”阶段。

## V4：增强功能（进行中）

目标：在不破坏现有 PWA 稳定性的前提下，增加长期使用所需的个性化、辅助决策和提醒能力。

推荐拆分顺序：
- V4.1：每日目标设置。
- V4.2：按时间进度判断补水建议。
- V4.3：历史统计增强。
- V4.4：自定义记录和自定义常用项目。
- V4.5：提醒功能研究与最小实现。
- V4.6：家人使用体验优化。

规划原则：
- 优先做不破坏 `records` 结构的功能。
- 优先使用独立 localStorage key 保存设置类数据。
- 先保持 `hydration_tracker_v3_records` 和 `schema_version: 1` 不变。
- 若以后需要扩展 JSON 备份，优先升级 `export_version`，而不是先改 `records` 结构。
- 旧数据清理仍为未来可选增强，不作为 V4 起点。

### V4.1：每日目标设置（已完成）

目标：允许不同用户使用不同的每日目标，不再固定为 `2000 ml`。

已实现要点：
- 默认目标仍为 `2000 ml`。
- 用户可设置如 `1500`、`1800`、`2000`、`2500 ml` 等目标，也可输入合理范围内的自定义值。
- 目标设置保存在本机 localStorage 的 `hydration_tracker_v4_settings`。
- 今日概览完成率使用用户设置目标。
- 历史月历完成率使用用户设置目标。
- 近 7 日趋势目标线使用用户设置目标。
- 不修改 `hydration_tracker_v3_records` 中已有 `records` 结构。
- `schema_version` 继续保持 `1`。
- JSON 导入导出格式本轮不升级，仍只迁移 records。

### V4.2：按时间进度判断补水建议

目标：根据当前时间和每日目标，判断当前补水进度是否落后。

规划要点：
- 默认饮水时间段可先设为 `08:00-22:00`。
- 理论应摄入量根据时间段线性估算。
- 如果落后，显示建议补水量。
- 建议与 V4.1 共用设置容器，而不是写入 `records`。
- 该功能会影响今日页文案和提示逻辑，但不必改历史记录结构。

### V4.3：历史统计增强

目标：提供比近 7 日趋势更清晰的长期观察能力。

规划要点：
- 可增加 7 日、30 日平均值和连续达标天数。
- 可增加按周或按月汇总。
- 优先从现有 `records` 派生统计，不新增记录字段。
- 若仅是只读统计，通常不需要新 key，也不需要 `schema_version` 升级。

### V4.4：自定义记录和自定义常用项目

目标：支持记录不在内置库中的食物和饮品，并允许逐步沉淀为个人常用项目。

规划要点：
- 自定义记录可先复用现有记录字段保存名称、单位、数量、估算水分和备注。
- 个人自定义常用项目建议放入独立 key，例如 `hydration_tracker_v4_custom_items`。
- 如果只增加自定义项目库，一般不必改已有 `records`。
- 若后续为记录增加明确的 `custom_item_id` 或更复杂元数据，才需要评估 `schema_version` 升级。

### V4.5：提醒功能

目标：在适当时间提醒补水。

规划要点：
- 这是 V4 风险最高的功能之一。
- iPhone Safari 和主屏幕 Web App 的提醒能力、权限、前台后台限制需要单独验证。
- 不应在未验证平台限制前承诺稳定提醒。
- 建议先做能力研究，再决定是否提供最小提醒实现。

### V4.6：家人使用体验优化

目标：让非技术用户更容易上手和迁移数据。

规划要点：
- 可优化首次打开引导、备份说明和导入提示。
- 可优化“换手机如何迁移”的文案与流程。
- 这类功能多数不需要改 `records` 结构。
- 优先做体验层和文档层改进，再考虑更多个性化设置。

## 当前阶段

V1、V2、V3 均已完成并正式收尾。当前项目已进入 V4，并完成 V4.1 每日目标设置。

近期推荐的默认路线是：
1. 下一步做 V4.2 时间进度补水建议。
2. 之后做 V4.3 历史统计增强。
3. 再评估 V4.4 自定义记录和自定义常用项目。
4. 提醒和家人体验优化继续后置。
