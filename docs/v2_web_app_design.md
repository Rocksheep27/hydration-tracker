- 通过只读 `GET /api/today` 返回今日汇总。
- 显示今日总水分、默认目标完成率、剩余量和今日记录数量。
- 复用 V1 的 JSON 读取、记录筛选和计算逻辑。
- 保持 V1 命令行版本可运行。
- V2.1 不提供新增、删除、清空或其他数据修改能力。

V2.2 网页添加记录已完成，并已在 Mac 本地浏览器中手动试用通过：

- 常用项目通过 `GET /api/common-items` 复用 V1 项目库。
- 用户可以输入数量、选择餐次并填写可选备注。
- 页面显示默认份量和本次水分估算。
- `POST /api/records` 在后端重新校验并计算记录。
- 保存成功后自动刷新今日概览。
- 数据只写入本地 JSON，不联网、不上传。

V2.2 不包含：

- 自定义项目或精确 `g/ml` 网页输入。
- 修改或删除单条记录。
- 清空今日记录。
- 提醒、账号、云同步、历史趋势或每日目标设置。

V2.3 网页记录明细与整理操作已完成，并已在 Mac 本地浏览器中手动试用通过：

- `GET /api/records/today` 返回今天的记录列表。
- 页面显示时间、类型、餐次、名称、原始数量和单位、估算水分及备注。
- `DELETE /api/records/{id}` 只按 `id` 删除今天对应记录。
- `DELETE /api/records/today` 只清空今天的数据并返回删除数量。
- 单条删除和清空今日都要求浏览器二次确认。
- 操作成功后自动刷新今日概览和记录明细。
- 历史日期记录不会被今日删除接口修改。

V2.3 不包含：

- 修改已有记录。
- 提醒、账号、云同步或历史趋势。
- iPhone Safari 专项适配、主屏幕安装或 iOS 原生 App。
- 自定义每日目标。

V2.1、V2.2 和 V2.3 均已完成，V2 本地 Web App 阶段可以正式收尾。下一阶段是 V3：适配 iPhone Safari。本文其余内容继续作为 V2 后续维护和验证依据。

## V2 目标

V2 要把 V1 已验证的核心功能做成浏览器界面，降低日常输入和核对记录的操作成本。

具体目标：

- 先在 Mac 本地运行。
- 提供适合普通用户的浏览器界面。
- 复用 V1 的项目库、计算规则和本地 JSON 数据。
- 保持计算逻辑、数据读写和界面展示分开。
- 为 V3 适配 iPhone Safari 保留清晰、可迁移的结构。

V2 暂不实现：

- 系统提醒或浏览器通知。
- 账号和登录。
- 云同步或数据上传。
- iOS 原生 App、Swift 和 App Store 配置。
- iPhone Safari 专项适配和添加到主屏幕。
- 自定义每日目标、历史趋势等增强功能。

每日目标继续使用 V1 默认值 `2000 ml`。

## 最小页面流程

### 页面一：今日概览

目的：打开应用后快速了解今天的摄入情况。

显示内容：

- 今日总水分摄入。
- 目标完成率。
- 距离默认目标还差多少。
- 今日记录数量。
- “添加记录”入口。
- “查看明细”入口。

页面流程：

1. 页面加载后请求今天的汇总数据。
2. 用户选择“添加记录”进入添加页面。
3. 用户选择“查看明细”进入今日记录明细页面。

### 页面二：添加记录

目的：以最少步骤添加一条记录。

输入流程：

1. 选择类型：白水、饮品或食物。
2. 从对应的常用项目中选择名称，也可以进入精确输入方式。
3. 输入数量。
4. 常用项目自动带出默认单位、估算重量或容量及含水率。
5. 页面即时显示本次估算水分，并明确标注为估算值。
6. 用户确认后保存记录。
7. 保存成功后返回今日概览，并刷新今日进度。

校验要求：

- 数量必须是大于 0 的数字。
- 精确输入必须使用后端支持的单位和有效含水率。
- 浏览器端可以即时提示，但后端仍需再次校验，不能只依赖 JavaScript。
- 保存失败时保留用户已输入的内容，并显示简短错误信息。

### 页面三：今日记录明细

目的：逐条核对今天的记录并处理误操作。

显示内容：

- 时间。
- 名称。
- 用户输入的原始数量和单位。
- 估算水分。
- 必要时显示换算后的重量或容量和含水率。

可用操作：

- 删除单条记录：按记录 `id` 删除，操作前弹出二次确认。
- 清空今日记录：明确提示影响范围，二次确认后只删除今天的数据。
- 返回今日概览：删除或清空后重新获取汇总数据。

删除操作不得使用页面序号作为永久标识，以免列表变化后误删；页面序号只用于展示，实际请求使用记录 `id`。

## 技术路线比较

### 方案 A：Python 标准库 `http.server` + HTML/CSS/JavaScript

实现方式：

- 使用 `http.server` 中的 HTTP 服务类建立本地服务器。
- 编写一个小型请求处理器，提供静态文件和少量 JSON API。
- 浏览器使用原生 `fetch` 调用 API。
- 直接导入 V1 的 Python 模块复用计算和 JSON 逻辑。

优点：

- 不需要安装第三方依赖。
- 与当前 V1 的标准库方向一致。
- 文件少，运行过程容易观察。
- 适合验证本地浏览器最小流程。

限制：

- 路由、请求体解析和错误响应需要自己写少量代码。
- 不适合复杂网站或公开部署。
- 必须严格限制功能范围，避免逐步写成难维护的自制框架。

### 方案 B：Flask 小型 Web App

实现方式：

- 使用 Flask 提供路由、请求解析、静态文件和模板能力。
- Python 路由直接调用 V1 核心函数。

优点：

- 路由和 JSON API 写法更清楚。
- 错误处理、模板和后续扩展更方便。
- Web 项目结构更常见，学习资料较多。

限制：

- Flask 是第三方依赖，需要单独安装和记录版本。
- 会增加虚拟环境和依赖管理步骤。
- 对当前只有三个页面的本地原型来说，暂时不是必需。

### 推荐方案

V2 最小版本推荐 **方案 A**。

原因：当前功能范围小、只在 Mac 本地运行，并且 V1 已完全使用 Python 标准库。基于 `http.server` 增加一个很小的本地 JSON API，可以在不安装依赖的前提下完成浏览器流程验证。

实施边界：

- 服务器只监听 `127.0.0.1`，不作为公开网站使用。
- API 数量保持最小，不自行实现通用框架。
- 如果后续出现复杂路由、模板、会话或更多数据页面，再重新评估 Flask。
- Flask 如被采用，需要先明确安装命令和版本，但本轮不安装。

## 最小 API 边界

建议由 `web_server.py` 提供同源 JSON API：

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/api/common-items` | 返回常用食物和饮品及默认估算值。 |
| `GET` | `/api/today` | 返回今日总量、完成率、剩余量和记录数量。 |
| `GET` | `/api/records/today` | 返回今日记录明细。 |
| `POST` | `/api/records` | 校验并保存一条新记录。 |
| `DELETE` | `/api/records/{id}` | 按记录 `id` 删除一条今日记录。 |
| `DELETE` | `/api/records/today` | 二次确认后清空今日记录。 |

页面和 API 由同一个本地服务器提供，不需要跨域请求，也不需要联网。

## 复用 V1 逻辑

V2 的服务器层应导入 `hydration_tracker.py`，不要在 JavaScript 中复制计算规则。

### 常用食物和饮品库

- 复用 `COMMON_ITEM_DEFAULTS` 和项目分组数据。
- `/api/common-items` 把这些数据转换为 JSON 返回给浏览器。
- 浏览器只负责展示，默认重量、容量和含水率以 Python 数据为准。

### 水分估算逻辑

- 常用项目复用 `create_common_item_record()`。
- 精确输入复用 `create_record()` 和 `calculate_estimated_water_ml()`。
- API 保存前必须调用 Python 逻辑校验，不能直接相信浏览器提交的估算结果。

### 今日总量和进度

- 复用 `calculate_daily_total()`。
- 复用 `calculate_completion_rate()`。
- 复用 `calculate_remaining_ml()`。
- 今日记录数量通过 `get_records_for_date()` 计算。

### JSON 读写和记录管理

- 复用 `load_records()`、`save_records()` 和 `add_record()`。
- 复用 `get_records_for_date()` 和 `clear_records_for_date()`。
- V1 的 `delete_record_for_date()` 继续按今日序号服务命令行。V2.3 在 Web 服务器层按日期和记录 `id` 删除，未改变 V1 原函数和命令行行为。
- API 测试必须传入临时数据文件，不能使用正式 `data/hydration_log.json`。

### 不复用的部分

- `input()`、`print()` 和命令行菜单函数只服务于 V1 终端界面。
- Web 页面不调用命令行提示函数。
- V2 只复用数据、计算和存储函数，界面状态由 HTML 和 JavaScript 管理。

## 风险和注意事项

### 保持 V1 可用

- 不删除或替换 V1 命令行入口。
- 每次 V2 改动后继续运行现有 V1 测试。
- 定期确认 `python3 src/hydration_tracker.py` 仍能启动。
- 新的服务器入口单独放在 `src/web_server.py`。

### 保护本地数据

- 开发和自动测试使用临时 JSON 文件。
- 不清空、不迁移、不覆盖现有用户数据。
- 保持现有 JSON 字段向后兼容。
- 删除接口必须按记录 `id` 操作，并限制为今天的记录。
- 清空接口必须再次检查目标日期是今天。

### 本地和隐私边界

- V2 默认只监听 `127.0.0.1`。
- 不调用外部 API，不加载外部脚本、字体或统计服务。
- 所有数据继续保存在本机，不上传。
- V3 若需要从 iPhone 访问 Mac，再单独评估局域网访问和安全边界。

### Web 输入安全

- 后端验证类别、数量、单位、含水率和记录 `id`。
- 页面显示名称和备注时使用文本方式插入，避免把用户内容当作 HTML。
- 静态文件服务必须限制在 `web/` 目录，防止读取项目外文件。
- 返回统一、简短的 JSON 错误信息，不向页面暴露本机路径或内部异常。

### 标准库服务器边界

- 只用于单用户、本机、开发阶段。
- 初期使用顺序处理请求，避免同时写入 JSON 导致覆盖。
- 不将该服务器直接暴露到互联网。
- 如果 V2 范围扩大，应停止增加自制路由并重新评估 Flask。

## 验证要求

- 原有 V1 测试继续全部通过。
- 为 API 使用临时 JSON 文件编写测试。
- 手动验证三个页面之间可以正常往返。
- 验证添加后概览和明细同步更新。
- 验证删除和清空需要二次确认，并且不影响历史日期。
- 验证服务器停止后，数据仍保存在本地 JSON。
