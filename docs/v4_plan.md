# V4 Plan

## 目标

V4 不是重做 PWA，而是在当前已稳定发布的 V3 基础上，逐步增加个性化、辅助决策和提醒能力，同时保持以下边界不变：

- 不删除或覆盖用户已有记录。
- 不调用 `localStorage.clear()`。
- 不让 Service Worker 读取、缓存或修改用户记录数据。
- 不破坏已有 JSON 备份。
- 不让旧备份失效。
- 不引入账号系统、云同步或 GitHub 端数据存储。

## 功能优先级

### 第一优先级

#### V4.1 每日目标设置

推荐最先实现。

原因：
- 当前默认目标固定为 `2000 ml`，这是最明显的个性化缺口。
- 目标值会直接影响今日概览、完成率、历史月历和近 7 日趋势，是后续建议和统计功能的基础。
- 该功能可以通过独立设置容器实现，风险明显低于提醒和自定义记录。

### 第二优先级

#### V4.2 按时间进度判断补水建议

原因：
- 产品价值高，能把“记录工具”进一步变成“辅助决策工具”。
- 与 V4.1 强相关，需要先拿到用户的每日目标。
- 主要依赖计算和提示逻辑，理论上不必改 `records` 结构。

#### V4.3 历史统计增强

原因：
- 当前已有月历和近 7 日趋势，但缺少更长期、更结构化的统计视角。
- 该功能大多可以从现有 `records` 直接派生，风险较低。

### 第三优先级

#### V4.4 自定义记录 / 自定义常用项目

原因：
- 用户价值高，但会明显增加数据边界和输入复杂度。
- 需要仔细设计“记录字段”“个人项目库”“备份迁移”“旧记录兼容”等问题。

#### V4.6 家人使用体验优化

原因：
- 正式发布后会逐渐变得重要。
- 多数属于流程、文案、引导和迁移体验优化，可以在主要功能稳定后逐步补充。

### 最后优先级

#### V4.5 提醒功能

原因：
- 浏览器提醒能力在 iPhone 上限制较多，真实可用性最不确定。
- 这类功能一旦实现不稳定，反而更容易破坏用户信任。
- 应先做平台能力研究和最小验证，再决定是否真正进入实现阶段。

## 数据结构与存储规划

### 现有稳定数据边界

当前运行时边界：
- `hydration_tracker_v3_records`：保存 `{ schema_version, records }`
- `schema_version = 1`
- JSON 备份格式当前以 `records` 为核心，兼容现有导入导出流程

结论：
- 在没有充分理由前，不应修改 `hydration_tracker_v3_records.records` 中既有记录结构。
- 设置类能力优先走独立 key，而不是往 `records` 里混入新的全局配置。

### 推荐新增 key

#### `hydration_tracker_v4_settings`

建议给 V4.1 及后续设置类能力使用。

建议承载内容：
- `daily_target_ml`
- 未来可扩展：`hydration_window_start`、`hydration_window_end`
- 未来可扩展：提醒开关、显示偏好、家庭模式引导状态

推荐原因：
- 与记录数据职责分离。
- 不影响既有 `records` 结构。
- 便于以后单独校验、迁移和备份。

#### `hydration_tracker_v4_custom_items`

建议留给 V4.4 自定义常用项目。

建议承载内容：
- 自定义项目列表
- 每个项目的 `id`、`name`、`category`、`default_unit`、`default_amount`、`base_type`、`water_ratio`、`emoji`

推荐原因：
- 个人项目库不应和记录容器混在一起。
- 可以单独校验，不污染内置项目库。

## 哪些功能会影响数据结构

### 不必改变现有 `records` 结构的功能

- V4.1 每日目标设置
- V4.2 时间进度补水建议
- V4.3 历史统计增强
- V4.6 家人使用体验优化

这些功能更适合：
- 读取现有 `records`
- 使用独立设置 key
- 在界面层和计算层增加能力

### 可能影响数据结构的功能

- V4.4 自定义记录 / 自定义常用项目
- V4.5 提醒功能中的高级状态持久化

说明：
- 自定义记录如果只复用现有记录字段，未必需要改 `schema_version`。
- 但如果后续要求记录明确保存 `custom_item_id`、自定义计算方式、来源标记或更复杂元数据，就要重新评估记录结构。
- 提醒若只保存提醒设置，放在 `hydration_tracker_v4_settings` 即可；若要保存复杂提醒状态，再考虑是否需要更多结构。

## `schema_version` 升级建议

### 当前建议

V4.1 不建议升级 `schema_version`。

原因：
- 每日目标设置不要求修改 `records`。
- 目标值是全局设置，不属于单条记录的一部分。
- 单独设置 key 已足够支撑 V4.1。

### 未来才可能触发升级的情况

只有在以下情况出现时，才建议评估把 `schema_version` 从 `1` 升级：
- 单条记录必须新增运行时必需字段。
- 旧记录在新功能下无法被正确解释。
- 现有导入器无法仅靠默认值或回退规则兼容旧记录。

例如：
- 每条记录新增不可缺失的 `source_type`
- 每条记录新增必须参与计算的 `target_snapshot`
- 记录结构从单数组改为分层容器

当前 V4 规划里，没有任何功能必须在第一步就做这些改动。

## JSON 导入导出兼容性规划

### V4.1 是否要立刻升级 JSON 备份格式

推荐结论：
- 运行时先新增 `hydration_tracker_v4_settings`。
- V4.1 已按第一阶段方案落地，当前不修改 JSON 备份格式。
- 但应在规划上预留后续兼容升级方案。

原因：
- 这样可以先最小风险交付“每日目标设置”。
- 不会影响现有 JSON 备份导入。
- 不会让旧备份失效。

代价：
- 如果只导出 `records`，换设备后目标设置不会跟着迁移。

### 推荐的后续兼容方案

在 V4.1 稳定后，再评估把 JSON 备份升级到 `export_version = 2`。

推荐方向：
- 继续保留现有 `records` 字段。
- 在导出顶层新增可选 `settings` 对象。
- 导入器同时支持：
  - `export_version = 1`：只有 `records`
  - `export_version = 2`：`records + settings`

兼容原则：
- 旧备份仍可导入。
- 新备份导入到旧版本页面时，可以明确提示版本不支持，而不是静默失败。
- 如果导入的是旧备份，没有 `settings`，则保留本机当前设置，或在无设置时退回默认 `2000 ml`。

### 如何保证旧备份仍可导入

必须满足：
- 不改变旧备份中 `records` 的必需字段含义。
- 导入器继续接受 `export_version = 1`。
- `schema_version = 1` 的记录仍按现有方式解析。
- 新功能依赖的设置必须有默认值，不能要求旧备份必须提供。

## 各小版本拆分

### V4.1 每日目标设置（已完成）

目标：
- 让用户设置自己的每日目标。
- 让今日页、历史月历和近 7 日趋势统一使用用户目标。

已实现数据边界：
- 新增 `hydration_tracker_v4_settings`。
- 设置对象为 `{ settings_version: 1, daily_goal_ml }`。
- 界面只保留自定义每日目标输入，不保留固定快捷目标选项。
- 保存交互以显式点击保存按钮为准，不再依赖已移除快捷选项的旧交互路径。
- 不修改 `hydration_tracker_v3_records`。
- 不升级 `schema_version`。
- 第一阶段不升级 JSON 备份格式。
- 当前历史日期完成率统一按当前目标重新计算，不保留按日期目标快照。

测试重点：
- 默认目标仍为 `2000 ml`。
- 设置新目标后，今日完成率即时更新。
- 历史月历完成率随目标变化更新。
- 近 7 日趋势目标线随目标变化更新。
- 刷新页面后设置仍保留。
- 旧 `records` 不变。
- 旧 JSON 备份仍可导入。

### V4.2 时间进度补水建议

目标：
- 根据当前时间和目标，提示理论应摄入量、是否落后以及建议补水量。

数据边界：
- 复用 `records`。
- 复用或扩展 `hydration_tracker_v4_settings`。
- 不修改 `records` 结构。
- 正常情况下不升级 `schema_version`。

测试重点：
- 不同时间点的理论应摄入量计算正确。
- 落后时提示合理，领先时不误报。
- 默认时间段与自定义时间段回退逻辑清楚。
- 今日页刷新和重开后结果稳定。

### V4.3 历史统计增强

目标：
- 提供更长期的趋势和统计摘要。

数据边界：
- 从现有 `records` 派生统计。
- 除非确有必要，不新增持久化结构。
- 不修改 `records`。

测试重点：
- 7 日、30 日、按周或按月统计正确。
- 空数据和跨月场景正确。
- 历史页性能在本地数据量增长后仍可接受。

### V4.4 自定义记录 / 自定义常用项目

目标：
- 支持库外食物和饮品。
- 支持沉淀为个人常用项目。

数据边界：
- 自定义项目建议放入 `hydration_tracker_v4_custom_items`。
- 记录可先复用现有字段；是否升级 `schema_version` 取决于记录层是否新增必需字段。
- JSON 备份大概率需要扩展到 `export_version = 2` 或更高，以便一起迁移个人设置和自定义项目。

测试重点：
- 自定义记录计算与保存正确。
- 自定义项目不会破坏内置项目库。
- 旧记录显示兼容。
- 新旧备份导入行为清楚。

### V4.5 提醒功能

目标：
- 在平台能力允许的前提下提供最小提醒能力。

数据边界：
- 提醒设置优先放入 `hydration_tracker_v4_settings`。
- 不让 Service Worker 操作用户记录数据。
- 不引入云同步。

测试重点：
- iPhone Safari 与主屏幕模式下权限行为是否可用。
- 用户拒绝权限后的回退体验。
- 提醒不影响记录保存和离线使用。

### V4.6 家人使用体验优化

目标：
- 降低非技术用户的首次使用和迁移门槛。

数据边界：
- 优先不改 `records`。
- 轻量偏好可放入 `hydration_tracker_v4_settings`。

测试重点：
- 新用户首次进入是否能理解三视图。
- 备份迁移说明是否清楚。
- 家人设备上从导入到开始使用的流程是否顺畅。

## 建议延后的功能

优先延后的功能：
- 提醒功能
- 复杂自定义记录元数据
- 批量旧数据清理

原因：
- 平台限制多，风险高，最容易引入兼容问题或误操作。
- 当前 PWA 已能长期使用，优先补足个性化和统计价值更合理。

## 本轮不做

- 不做业务功能代码。
- 不修改 `pwa/app.js`、`pwa/index.html`、`pwa/styles.css` 或 `pwa/service-worker.js`。
- 不修改 `hydration_tracker_v3_records` 结构。
- 不升级 `schema_version`。
- 不升级 JSON 导入导出实现。
- 不做账号、云同步或 GitHub 端数据存储。
